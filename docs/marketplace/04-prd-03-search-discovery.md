# PRD-03 — Search & Discovery

## Current search/filter capabilities (code-backed)

The system has **browse and lookup, not search**:

1. **Alphabetical browse** (`index.md`, generated by `scripts/generate_index.py` / `scripts/rebuild_index.py`): letter-bucketed tables (`0-9`, `A`–`Z`) with a jump bar; columns are Game, Players, Time, Designer; expansions nest under base games with `↳`.
2. **Direct URL lookup**: the published contract `https://jonnyallred.github.io/boardgame-rules/rules/{slug}/` — clients (including AI assistants, per the landing-page prompt) compute the slug themselves. Pretty permalinks via `_config.yml: permalink: pretty`.
3. **Grep/Glob lookup** (agent-side): the `rules-lookup` skill finds games via `Glob rules/<game>*.md`, falls back to `games.yaml` name match, and answers questions by Grepping keywords inside the summary, then the extracted text, then BGG forums (`site:boardgamegeek.com` WebSearch), then labeled intuition.
4. **Registry queries** (operational): `get_games_by_status()` filters by pipeline status — operator-facing only.

There is **no full-text index, no facet engine, no query API**. Jekyll serves static pages; any search today happens client-side in the consumer (human ctrl-F or LLM fetch-and-read).

## How the catalog hierarchy enables or constrains search

**Enables:**
- Stable `bgg_id` keys mean search results, listings, and rules pages all join cleanly; zero duplicate ids verified across 2,122 entries.
- Fixed section headings in every rules file (8 base / 6 expansion, machine-validated) make **section-scoped search** possible — e.g. search only Components sections to find games with specific pieces, or only Player Reference for quick-rule lookups. The `rules-lookup` skill already exploits this structure manually.
- Frontmatter is uniform and machine-checked (`scripts/validate.py`), so a crawler/indexer can trust it.

**Constrains:**
- **No edition layer** → search can't distinguish printings; a marketplace query like "Concordia, latest printing, German" is inexpressible (see PRD-01).
- **Flat files** → every "query" is a full scan or a pre-generated page. The regenerate-the-whole-index pattern works at 1,649 games for one page but will not extend to faceted, per-query marketplace search; an actual search index (SQLite FTS, Postgres, or a hosted engine) is a prerequisite. [NEEDS REVIEW: choose alongside the PRD-02 storage migration]
- **Slug-as-API** → any search system must preserve existing slugs as permalinks; 1,649 published URLs are effectively frozen.
- **Name-keyed writes** (`update_game` matches case-insensitive name) mean renames are hazardous — search-driven features that surface rename suggestions must update name, slug, and file path coherently or break the join. [NEEDS REVIEW]
