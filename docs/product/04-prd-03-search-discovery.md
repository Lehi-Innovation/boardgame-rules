# PRD-03 — Search, Discovery & AI Consumption

## Current capabilities (code-backed)

The system has **browse and lookup, not search**:

1. **Alphabetical browse** (`index.md`, generated by `scripts/generate_index.py` / `scripts/rebuild_index.py`): letter-bucketed tables (`0-9`, `A`–`Z`) with a jump bar; columns are Game, Players, Time, Designer; expansions nest under base games with `↳`.
2. **Direct URL lookup**: the published contract `https://jonnyallred.github.io/boardgame-rules/rules/{slug}/` — clients (including AI assistants, per the landing-page prompt) compute the slug themselves. Pretty permalinks via `_config.yml: permalink: pretty`.
3. **Grep/Glob lookup (agent-side)**: the `rules-lookup` skill finds games via `Glob rules/<game>*.md`, falls back to `games.yaml` name matching, and answers questions by Grepping keywords inside the summary, then the extracted text.
4. **Registry queries (operational)**: `get_games_by_status()` filters by pipeline status — operator-facing only.

There is **no full-text index, no facet engine, no query API**. Jekyll serves static pages; any search happens client-side in the consumer (human Ctrl-F or LLM fetch-and-read).

## The AI consumption contract

This is the product's primary "search" interface and deserves explicit treatment [code-backed, `index.md`]:

- The landing page ships a copy-paste system prompt that teaches any assistant to (a) compute `{slug}` from a game name, (b) fetch `rules/{slug}/`, and (c) fall back to the index page for the full game list.
- The same contract powers voice usage: paste the prompt in the Claude mobile app, switch to voice mode, ask questions hands-free at the table.
- Implications already honored by the code: slugs are deterministic (`slugify()`), URLs are stable, frontmatter and section headings are machine-validated, expansions resolve to their own pages.
- Implication **not** yet honored: there is no machine-readable index (JSON/sitemap) — assistants must parse the human-oriented Markdown tables to enumerate games. [OPEN QUESTION: publish a `games.json` alongside `index.md`?]

## Tiered deep lookup (`/rules` command)

For questions a summary can't answer, `.claude/skills/rules-lookup/SKILL.md` defines a 4-tier escalation, stopping at the first tier that yields a clear answer:

1. **Summary file** (`rules/<slug>.md`) — answer and stop, no update needed.
2. **Extracted rulebook** (`extracted/<slug>*.txt`) — answer, then update the summary.
3. **BGG forum search** (`site:boardgamegeek.com` WebSearch) — official rulings or strong consensus, with URL citation.
4. **Intuition** — best-guess from game conventions, explicitly labeled "unverified."

Every answer states its source tier and a confidence level (high/medium/low). Tiers 2–4 trigger an automatic summary update and PR (`rules-update/<game>-<short-slug>` branches), making search usage feed content quality.

## How the catalog hierarchy enables or constrains search

**Enables:** stable `bgg_id` joins; **section-scoped search** thanks to the fixed validated headings (e.g., search only Components sections, or only Player Reference for quick rules) — the `rules-lookup` skill already exploits this structure manually; trustworthy frontmatter for any future indexer.

**Constrains:**
- **Flat files** → every query is a full scan or a pre-generated page. Fine at 1,649 games for one index page; an actual search index (SQLite FTS or a static search library like Lunr/Pagefind) would be needed for full-text site search. [OPEN QUESTION: is full-text site search wanted, given AI assistants do retrieval for the primary audience?]
- **Slug-as-API** → 1,649 published URLs are effectively frozen; renames are breaking changes.
- **Name-keyed writes** (`update_game` matches case-insensitive name) make renames hazardous: name, slug, file path, and index entry must move together or the join breaks. [NEEDS REVIEW: rename procedure or tooling]
